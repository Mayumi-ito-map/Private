# 記号ゆれ「最初」と「今回」の違い — 範囲を狭め過ぎたか

## 違いの整理

| 時期 | QUOTE_LIKE_CHARS | 結果 |
|------|------------------|------|
| **最初**（記号ゆれを作ったとき） | 7文字（' ' ` ʻ ʿ ´ ʼ） | 候補が約2000増えた。行数（rows_with_hit / rows_still_0）は Base と同数。 |
| **今回**（symbol_summary を反映後） | 2文字（' '）のみ | Base と**全く同じ**（候補増もなし）。 |

## 何が起きていたか

- **symbol_summary.xlsx** は **excel_local の「修正後欧文地名」だけ** から集計しています。**略語変換表.xlsx は集計対象にしていません。**
- **司令塔①** は「修正後欧文地名」に **略語変換表の置き換え**（略称→フルスペル）をかけます。なので **normalized_name に含まれる文字は、次の2つだけが元**です。
  1. **excel_local の「修正後欧文地名」**（元Excel）
  2. **略語変換表の「フルスペル」**（略語を置き換えたときに使う文字列）
- したがって **「司令塔①の処理で、元Excelにも略語変換表にも無い文字が新たに加わる」ことはありません。** 出てくる文字は、必ずどちらか（または両方）に含まれている文字だけです。
- ただし **symbol_summary は excel_local しか見ていない** ので、**略語変換表のフルスペルにだけ ʻ や ʿ などが含まれている場合**、normalized_name にはその文字が出てきても、symbol_summary には出てきません。  
  → その結果、「symbol_summary では ' と ' しか出てこなかった」のに、**実際の normalized_name には ʻ や ʿ が（略語変換表経由で）含まれていた**、ということはあり得ます。
- **2文字に絞ったとき**は、ʻ ʿ 等を対象外にしたので、**それらが含まれる候補**では記号ゆれの展開が行われず、候補が増えなかったと考えられます。
- **最初の7文字のとき**は ʻ ʿ 等も対象だったので、**略語変換表由来で ʻ 等が入っている候補**でも展開が行われ、候補が約2000増えていた、という整理です。

つまり、

- **司令塔①が「無い文字を勝手に足す」わけではありません。** 出てくる文字は「元Excel」か「略語変換表のフルスペル」のどちらか（または両方）にあります。
- **symbol_summary は「元Excelだけ」を集計している**ため、**略語変換表にだけある文字（ʻ ʿ 等）は symbol_summary に現れない**、という説明のずれがありました。
- **範囲を 2 文字に絞ったことで、略語変換表由来の ʻ 等が含まれる候補の展開が落ち、結果として「範囲を狭め過ぎた」**と言えます。

## 対応

- **QUOTE_LIKE_CHARS を再び 7 文字に戻しました**（' ' ` ʻ ʿ ´ ʼ）。  
  → 最初の記号ゆれを作ったときと同じ条件で、再度実行すると「候補が約2000増える」ような結果に戻る想定です。
- 行数（rows_with_hit / rows_still_0）は、最初のときと同様に Base と変わらない可能性はありますが、**候補数を増やす**（地図で距離を測るときに候補が多い方がよい）という目的には、7文字版の方が合っています。

## まとめ

- **最初**: 7文字 → 候補が増えた（約2000）。行数は Base と同数。
- **今回（2文字にしたとき）**: 候補増なし・Base と同一 → **範囲を狭め過ぎていた**。
- **今回（7文字に戻した後）**: 最初と同じく、記号ゆれで候補が増える動きに戻しています。
