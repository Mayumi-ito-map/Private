## 質問（1）候補は約2000増えたのに、rows_still_0 / rows_with_hit が同じだった理由

**「候補数」と「行数」は別物**だからです。

- **候補が約2000増えた**: 記号ゆれで **1行あたりの候補数** が増えた（例: 1候補→最大8候補）。その合計が約2000増。
- **rows_with_hit / rows_still_0 が同数**: どちらも **「行」単位** の集計。
  - ある行で **どれか1つでも候補がヒット** すれば、その行は `rows_with_hit` に 1 行だけカウントされる。
  - 追加した候補の多くは、
    - もともとヒットしていた行の「別表記」だった（行としては重複カウントにならない）、
    - または、もともとヒットしていない行で、記号を変えてもまだヒットしなかった（綴り・語順など別要因）、  
  のどちらかだったと考えられます。

その結果、「候補数」は増えても、「新たにヒットし始めた行」がほぼなかったため、**行数ベースの指標は変わらなかった**、という理解でよいです。  
一方で「都市が多く含まれた」「候補が多い方が地図で距離を測るのに有利」という効果は、**候補数が増えたこと**によるものなので、導入の判断としては妥当です。

## 質問（2）「,」「.」は司令塔①で扱うので、司令塔②の記号ゆれには含めない考え方でよいか

**その考え方で問題ありません。**

- カンマは司令塔①の **カンマ並び替え**、ピリオドは **略語（Mt. 等）** で既に扱っている。
- 司令塔②の記号ゆれは **アポストロフィ・引用符類**（' / ' / ʻ / ʿ など）の「表記ゆれ」に絞る設計でよいです。
- symbol_summary.xlsx で元データの記号を確認したうえで、「司令塔②では , . は対象にしない」と割り切ってよいです。

## 質問（3）「-」ハイフンはどう考えるか

**Stage2/3 では既に吸収されているので、記号ゆれで無理に増やさなくてよい**、という整理がおすすめです。

- **Stage2/3**: `_remove_symbols` で **ハイフン・ダッシュ類を削除** しているので、  
  "Saint-Martin" も "Saint Martin" も "SaintMartin" のように比較され、この時点ではハイフンの有無は原因になりません。
- **Stage1**: 空白・連続空白の正規化のみなので、  
  "Saint-Martin" と "Saint Martin" は別文字列のまま。ここだけハイフンが効く。

したがって:

- **記号ゆれの対象にハイフンを含めるか**  
  → 候補爆発（ハイフン→スペース／なし等）を避けるなら、**入れない**方針でよいと思います。
- **ハイフンで落ちているケース**が後から分かったら、  
  Stage1 用に「ハイフン↔スペース↔なし」の展開を **必要最小限のパターンだけ** 追加する、という段階的対応が現実的です。

## 質問（4）「quote展開は ' と ' だけで十分」について

**データの実態に合わせるなら、まずは ' と ' の2種から始めるのが妥当**だと思います。

- 地名データで実際に問題になるのは、多くの場合 **ASCII アポストロフィ ' (U+0027)** と **右シングルクォート ' (U+2019)** の混在。
- ʻ（オキナ）・ʿ（アイン）は **ハワイ語・アラビア語系** で重要だが、symbol_summary でほとんど出てこないなら、最初は対象外にしてよい。
- **' と ' だけ**にすると:
  - 候補は **最大 1+2＝3 種類** に抑えられる（質問（7）の「1+7」が「1+2」になる）。
  - ループ・辞書アクセスが減り、**実行時間の短縮**も期待できる。

**推奨**:  
まずは **QUOTE_LIKE_CHARS を [' , ' ] の2つに絞って試す**。  
中東・太平洋などで「明らかに ʻ/ʿ が原因の未マッチ」が出てきたら、その時点で ʻ / ʿ を追加する、という段階的な拡張がよいと思います。

## 質問（5）記号ゆれで約20分かかる理由と、記号のある国だけにする方法

**理由**  
記号ゆれで **1行あたりの候補数が最大で約8倍**になり、そのすべてに対して Stage1→2→3 の辞書引きをしているためです。

- 対象行数 × 候補数 × 3段階 のループが増える。
- 特に記号を含む地名が多い国（例: フランス語圏）で負荷が大きくなります。

**短縮の方向性**

1. **記号が含まれる行だけ展開する（現状の仕様を活かす）**  
   `expand_quote_variants` は、**文字列に QUOTE_LIKE_CHARS が1つも無ければ [text] のみ返す**実装になっています。  
   つまり「記号のない行はもともと展開していない」ので、さらに削るなら「記号のある行」に絞る処理を入れることになります。

2. **記号を含む行だけ展開するようにする（推奨）**  
   - 行の全候補をいったん `expand_candidates_quote_variants` に渡すのではなく、  
     **「その行の候補のどれかに、QUOTE_LIKE_CHARS が1つでも含まれるか」** を先に判定する。
   - **1つも含まれない行** → `expand_candidates_quote_variants` を呼ばず、元の `candidates` のまま Stage へ。
   - **1つでも含まれる行** → だけ `expand_candidates_quote_variants(candidates)` を実行。

   こうすると、記号のない国・行では追加の展開が入らないため、全体の実行時間をかなり短縮できます。

3. **記号のある国だけ実行する**  
   - 事前に「judge==0 の行の normalized_name に、QUOTE_LIKE_CHARS が含まれる国」だけリストアップするスクリプトを1本用意する。
   - 記号ゆれ付きの司令塔②は、**その国リストだけ**を対象に実行する。  
   これでも 20分→数分レベルに減らせる可能性があります。

方針が決まっていれば、「記号あり行だけ展開」を `run_stage_match_quote_variants.py` に組み込む形で具体的なパッチ案も出せます。

## 質問（6）記号ゆれも「ローカル対策」か／「候補を広げてから Stage で削除」の理解でよいか

**両方の理解で合っています。**

- **記号ゆれはローカル側の対策**  
  同義語展開と同様、**ローカル候補（normalized_name から得たリスト）** にだけ記号ゆれを適用しています。GeoNames 側の文字列は変更していません。

- **「司令塔②で、Stage1/2/3 で削除する前に、まず候補を広げる」**  
  流れは「**候補を広げる（記号ゆれ／同義語）→ その候補で Stage1/2/3 の正規化・マッチング**」です。  
  Stage の「削除」はマッチング時にキーを作るための正規化であり、その**前**に「ローカル側の候補を増やす」という意味で、ご理解の通りです。

## 質問（7）「最大1+7種類」の意味

**「1つの候補から、最大で 1（元の文字列）+ 7（記号を置き換えた7パターン）= 合計8種類まで候補が増える」** という意味です。

- 現在 `QUOTE_LIKE_CHARS` に **7文字**（' / ' / ` / ʻ / ʿ / ´ / ʼ）を定義している。
- ある候補に、このいずれかが **1つでも含まれている** とき:
  - **1**: 元の文字列をそのまま1つ。
  - **7**: 「文中の“記号類”を、それぞれの1文字にそろえた形」を1つずつ作る（重複は除く）。
- なので **最大 1 + 7 = 8 種類** まで、その候補から生成されます。
- 記号が含まれない候補は、**1種類のまま**（増えません）。

「' と ' だけ」にした場合は、**最大 1 + 2 = 3 種類** になります。

「記号あり行だけ展開」は、この動きを変えずに**速度だけ最適化する**オプションです。
