# 記号ゆれ展開で Base と結果が同じだった理由

## 結論

**司令塔②の対象（judge==0 の行）の `normalized_name` に、' (U+0027) や ' (U+2019) が含まれていない**可能性が高いです。  
記号ゆれは「その文字が含まれる候補だけ」候補を増やすため、**1文字も含まれていなければ候補は1つも増えず、Base と全く同じ動き**になります。

## 仕様の整理

- **記号ゆれの動き**: 各候補文字列に **QUOTE_LIKE_CHARS（' と '）が1つでも含まれる**ときだけ、その候補を「' にそろえた形」「' にそろえた形」を追加する。**含まれていなければ [元の1件] のまま**（expand_quote_variants の docstring の通り）。
- **司令塔②が扱う行**: 司令塔①の結果のうち **judge=="0"** の行だけ。
- **symbol_summary.xlsx**: **excel_local の「修正後欧文地名」**（司令塔①の入力）から集計した記号。つまり **全行**（judge 0/1/2+ すべて）で使われている記号。

## なぜ効いていないか

- **' や ' が含まれる地名**は、司令塔①の完全一致で既にマッチしている（judge=1 や 2+）ことが多い。
- そのため **judge==0 として司令塔②に回ってくる行**には、**' や ' がほとんど（または全く）含まれていない**可能性が高い。
- そうだと、記号ゆれは「展開する対象が1件もない」状態で、**候補リストは Base と完全に同一**になり、結果も同じになります。

つまり「記号ゆれのロジックが間違っている」のではなく、**今回のデータでは「記号ゆれの対象になる文字が、司令塔②の入力にほぼ出てこなかった」**と考えられます。

## 確認方法

次の診断スクリプトで、「judge==0 の行のうち、normalized_name に ' または ' が含まれる行が何件あるか」を確認できます。  
この数が 0 なら、記号ゆれが効かない理由の説明と一致します。
