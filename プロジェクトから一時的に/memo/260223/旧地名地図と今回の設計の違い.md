# 旧地名地図（http://52.194.188.168/）と今回の設計の違い

## 旧地名地図（見本）の構成（取得内容から分かること）

- **タイトル**: 地図データ表示
- **地域の選択**: 01_北海道 ～ 48_沖縄県（都道府県単位・日本国内のみ）
- **背景レイヤ**: 地理院淡色／標準／航空写真、今昔マップ（地域別）、OpenStreetMap、Googleマップ など

→ **1台のサーバに固定でデプロイされ、URL が常に http://52.194.188.168/ のように決まっている**前提の設計と考えられる。

---

## 設計の違い（なぜ見本では「index.json が見つからない」が起きないか）

### 1. デプロイの仕方（パスの決まり方）

| 項目 | 旧地名地図（52.194.188.168） | 今回（leaflet_map.html） |
|------|------------------------------|---------------------------|
| **動かし方** | サーバ上に置いてあるアプリを、固定 URL で開く | 各自の PC で `python -m http.server` を起動し、ブラウザで開く |
| **URL** | 常に `http://52.194.188.168/` など一定 | 起動したフォルダや開くパスで変わる（例: `.../leaflet_map.html` や `.../output_leaflet/leaflet_map.html`） |
| **相対パスの基準** | サーバの「ドキュメントルート」が固定なので、`index.json` や `data/xx.geojson` などは**常に同じパス**で解決される | 「今開いているページの URL」が基準。**どのフォルダでサーバを起動したか・どの URL で開いたか**で `index.json` の相対パスが変わる |

→ 見本は **「1つのサーバ・1つの場所」で動かす**前提なので、相対パスがずれない。  
今回の方式は **「誰が・どこで・どう開くか」** でパスが変わるため、index.json の場所と fetch のパスがずれると「見つからない」になる。

---

### 2. データの範囲と一覧の持ち方

| 項目 | 旧地名地図 | 今回 |
|------|------------|------|
| **対象** | 日本国内（都道府県 48） | 世界 193 国（cn002～cn516 など） |
| **一覧の出し方** | 都道府県リストが HTML や script に**直接書かれている**可能性が高い（48件なので埋め込みが現実的） | 国が 193 件あるため、**index.json で一覧を外出し**し、fetch で読み込む設計 |

→ 見本は一覧が **HTML/script に含まれている**と、fetch で index.json を読まないので、index.json のパス問題がそもそも発生しない。

---

### 3. 作業者ごとの環境の違い

| 項目 | 旧地名地図 | 今回 |
|------|------------|------|
| **環境** | 本番サーバ（52.194.188.168）は**1か所**。作業者の PC は「ブラウザでその URL を開くだけ」 | あなたの PC と作業者の PC で**別々に** `python -m http.server` を実行。プロジェクトの置き場所や、どのフォルダで起動するかが違うと、開く URL や相対パスが変わる |
| **再現性** | 「この URL を開けば同じように動く」 | 「output_leaflet でサーバを起動し、leaflet_map.html を開く」など、**手順を揃えないと** index.json のパスがずれる |

---

## まとめ

- **旧地名地図**は  
  - 固定サーバ・固定 URL で動かす  
  - 地域一覧は 48 件で、HTML/script に埋め込んでいる可能性が高い  
  → **fetch で index.json を読まない**（またはサーバ側でパスが固定されている）ため、相対パスで困らない。

- **今回の地図**は  
  - 国 193 件の一覧を **index.json に外出し**し、**fetch で読み込む**  
  - ローカルで「どのフォルダでサーバを起動したか」「どの URL で開いたか」で相対パスが変わる  
  → **開く URL と index.json の置き場所を一致させる必要**がある（例: output_leaflet でサーバ起動し、`/leaflet_map.html` で開く）。

**「見本では起きないが今回起きる」理由**は、  
- 見本: **サーバ固定＋一覧を fetch していない（または固定パス）**  
- 今回: **ローカル起動＋国一覧を fetch で読む＋パスが開き方に依存する**  
という設計の差によるものです。

---

## 今回の地図を「見本に近い挙動」にするには

1. **運用で揃える**  
   - 「必ず `cd output_leaflet` してから `python -m http.server 8000` し、`http://localhost:8000/leaflet_map.html` を開く」と手順を統一する。  
   - または、HTML 側で `BASE = new URL(".", window.location.href).href` のように「この HTML があるフォルダ」を基準にして index.json を読む（既に導入済み）。

2. **見本のように「一覧を fetch しない」ようにする**  
   - `build_leaflet_index.py` で **data_index.js**（例: `window.LEAFLET_COUNTRY_INDEX = [...]`）を出力し、  
   - leaflet_map.html で `<script src="data_index.js"></script>` で読み、国一覧は fetch ではなくこの変数を使う。  
   → index.json のパス問題は出なくなる（各国 GeoJSON の fetch は残る）。

3. **本番のように固定サーバで配る**  
   - 旧地名地図と同様、**1台のサーバに HTML と index.json と GeoJSON を置き、固定 URL で配る**。  
   → そのサーバ上では常に同じパスで解決されるので、相対パスで困らない。
