 # GeoNames データ構造再設計メモ

作成日: 2026-02-19

------------------------------------------------------------------------

## 1. 設計方針

GeoNames を「地名辞書」としてではなく、\
**地理オブジェクトデータベース（Geo Object DB）**として扱う。

目的は以下の将来フェーズへの対応：

-   段階的正規化マッチング（Stage1--3）
-   geonameid による一意管理
-   距離計算フェーズ（緯度経度使用）
-   国単位処理
-   feature class / feature code 別抽出
-   海外領土統合処理

------------------------------------------------------------------------

## 2. マスターデータ構造

### 入力

-   allCountries.txt（約1.77GB）

### 出力

-   単一マスター pickle ファイル

### 主キー

-   geonameid

### 基本構造

    by_id = {
      geonameid: {
        geonameid: str,
        name: str,
        asciiname: str,
        alternatenames: list[str],
        lat: float,
        lon: float,
        fcl: str,
        fcode: str,
        ccode: str,
        ad1: str,
        ad2: str,
        ad3: str,
        ad4: str,
        population: int,
        elevation: int,
        dem: int,
        timezone: str,
        mdate: str
      }
    }

### 注意事項

-   alternatenames は **カンマ区切り文字列 → list\[str\]
    に変換（必須）**
-   latitude / longitude は丸めず float で保持
-   19項目すべて保持（将来拡張を見越す）

------------------------------------------------------------------------

## 3. 副インデックス設計

検索高速化のため、副インデックスを同時生成する。

### by_ccode

    "JP" → [geonameid, ...]

### by_fcl

    "T" → [geonameid, ...]

### by_fcode

    "PPL" → [geonameid, ...]

### 意図

-   国単位抽出を高速化
-   自然地名（T/H/U/V）抽出
-   都市（PPL系）抽出
-   距離フェーズでの候補制限

------------------------------------------------------------------------

## 4. 海外領土処理

海外領土対応は別JSONにて管理。

例：

    {
      "AU": ["AU","CC","CX","HM","NF"],
      "NZ": ["NZ","TK"]
    }

処理時： - 本国コードから複数ccodeを取得 - by_ccode
を利用してIDを統合抽出

------------------------------------------------------------------------

## 5. 列名短縮設計

内部処理効率と可読性向上のため、短縮形を採用。

  元項目              短縮
  ------------------- -------
  latitude            lat
  longitude           lon
  feature_class       fcl
  feature_code        fcode
  country_code        ccode
  admin1 code         ad1
  admin2 code         ad2
  admin3 code         ad3
  admin4 code         ad4
  modification date   mdate

※ 意味は保持する。

------------------------------------------------------------------------

## 6. 将来フェーズへの接続

-   マッチング結果は geonameid で保持
-   正規化は name / asciiname / alternatenames に適用
-   GeoPandas DataFrame化を前提
-   距離計算は lat/lon を直接利用

------------------------------------------------------------------------

## 7. 構造の本質

設計思想：

-   by_id → 主キー（正体）
-   by_ccode / by_fcl / by_fcode → 副インデックス（検索用）

これは単なる辞書ではなく、 **検索可能な地理データベース構造である。**
