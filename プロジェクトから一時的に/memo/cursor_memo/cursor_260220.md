## 実施した改変のまとめ

### 1. 新規追加: `geonames_loader.py`
- **役割**: `geonames_master.pkl` の読み込みと、国コード（海外領土対応）でのフィルタを共通化。
- **主な関数**:
  - `load_master_pkl()`: pkl を読み `by_id` / `by_ccode` を返す。
  - `resolve_country_codes()`: `overseas_territories.json` を参照してマッチング対象の国コードリストを返す。
  - `get_records_by_country_codes()`: 指定国コード群のレコード一覧を取得。
  - `build_placename_dict()`: 完全一致用の「名前 → レコードリスト」辞書を構築。
  - `records_to_gdf()`: レコードから GeoDataFrame（Point ジオメトリ、EPSG:4326）を生成。次の段階の Leaflet・距離計測を想定。
  - `load_geonames_for_matching()`: 司令塔用の一括取得。返り値は `(placename_dict, records, gdf)`。

### 2. `run_local_build.py`（司令塔①）
- **廃止**: `output_json`（`*_dict.json`）への参照を削除。
- **データソース**: 上記 `geonames_loader` 経由で `geonames_master.pkl` のみを使用。
- **出力**: これまで通り `output_match/{base}_result.xlsx`、`world_stats.xlsx`。  
  完全一致で 1 件だけヒットした行には、次の段階用に `matched_geonameid` / `matched_lat` / `matched_lon` を追加。
- **その他**: 一時ファイルを除外（`~$` で始まるファイルは処理しない）。

### 3. `run_stage_match.py`（司令塔②）
- **廃止**: `output_json` を廃止し、`geonames_master.pkl` のみ使用。
- **参照先**: `normalizers.stage_matcher` の `normalize_stage1/2/3` を使用（`matchers` は使用しない）。
- **データ**: `load_geonames_for_matching()` で pkl からレコード取得し、`build_stage_maps(records)` で Stage1/2/3 のマップを構築。レコードは lat/lon 付き。
- **出力**: `output_match_results/{path.stem}.xlsx`（例: 入力が `cn002_AZ_アゼルバイジャン_result.xlsx` なら、出力も同じベース名の `cn002_AZ_アゼルバイジャン_result.xlsx`）、`world_summary.xlsx`。
- **world_summary.xlsx の列**: `country`, `total_0_target`, `stage1`, `stage1(%)`, `stage2`, `stage2(%)`, `stage3`, `stage3(%)`, `still_0`, `still_0(%)`（仕様書の「till_0(%)」は `still_0(%)` で統一）。

## 注意点・既知の点

1. **geonames_master.pkl の存在**  
   プロジェクトルートに `geonames_master.pkl` が必要です。まだない場合は `build_geonames_db.py` で `allCountries.txt` から生成してください。

2. **Excel の list 列**  
   `normalized_name` や `geonames_hits` は Python の list のため、Excel に書き出すと文字列化されます。司令塔②では `parse_normalized_name()` で `ast.literal_eval` してリストに復元しているため、そのままで問題ありません。

3. **司令塔②の入力ファイル名**  
   司令塔①の出力は `cn002_AZ_アゼルバイジャン_result.xlsx` のように、すでに `_result` が付いた名前です。司令塔②は `INPUT_DIR.glob("cn*_*.xlsx")` で読み、出力は `path.stem.xlsx` にしているため、`_result` が二重にはなりません。

4. **GeoPandas の利用箇所**  
   現状、GeoPandas は `geonames_loader.records_to_gdf()` および `load_geonames_for_matching()` の返り値 `gdf` で使用しています。司令塔①②のメイン処理では、まだ `gdf` をファイル出力や距離計算には使っていません。次の段階で Leaflet や距離計測を行う際に、この `gdf` や `matched_lat/lon` をそのまま利用できるようにしてあります。

5. **matchers/stage_matcher.py**  
   参照は `normalizers/stage_matcher.py` に統一済みです。`matchers/` を削除してよいかは、他で使っていなければ削除で問題ありません。

### 1) 司令塔②が少なく見える典型理由（今の実装に即して）
- **(A) 国コード縛りが強い**  
  `country_code`（＋海外領土）で絞った GeoNames にしか当てていません。  
  境界付近・越境河川/山脈・海域・島嶼などは、GeoNames 側の `ccode` が別国になっていることがあり、そこで落ちます。
- **(B) `normalized_name`（司令塔①）の候補生成が不足している**  
  司令塔②は `normalized_name` の候補だけを使います。ここに「本当に必要な表記ゆれ候補」が入っていないと、Stage を強めても当たりません。

  `reorder_by_comma()` はカンマがあると **並び替え後だけ**を返します（元文字列を候補に残していない）。  
  これが「元表記が実は GeoNames の alternate に存在する」ケースを落とす原因になりえます（略語展開では“元を残す”のに、カンマだけ残していない）。
- **(D) GeoNames 側にそもそも別表記が無い**  
  `alternatenames` に atlas のローマナイズが入っていない（あるいは別方式）だと、どれだけ正規化しても一致しません。
- **(E) 集計の見方の差**  
  司令塔②は「候補単位」集計だと見た目が厳しく見えます（行単位で“1候補でも当たればOK”という見方をすると印象が変わります）。
