# GeoNames データベース再設計メモ（技術設計版）

作成日: 2026-02-19

------------------------------------------------------------------------

## 1. 設計目的

GeoNames を単なる「地名辞書」として扱うのではなく、
**検索可能な地理オブジェクトデータベースとして再設計する。**

本設計は以下の将来フェーズを前提とする：

-   段階的正規化マッチング（Stage1--3）
-   geonameid を軸とした一意管理
-   距離計算フェーズ（緯度経度利用）
-   国単位・feature単位抽出
-   海外領土統合処理
-   GeoPandasによる空間処理

------------------------------------------------------------------------

## 2. 入力データ

-   allCountries.txt（約1.77GB）
-   全19列を保持（削減しない）

GeoNames 列インデックス：

    IDX = {
        "geonameid": 0,

        "admin4 code": 13,

        "dem": 16,

    }

------------------------------------------------------------------------

## 3. マスター構造（主キー設計）

### 主キー：geonameid

    by_id = {
      geonameid: {
        geonameid: str,
        name: str,
        asciiname: str,
        alternatenames: list[str],
        lat: float,
        lon: float,
        fcl: str,
        fcode: str,
        ccode: str,
        ad1: str,
        ad2: str,
        ad3: str,
        ad4: str,
        population: int,
        elevation: int,
        dem: int,
        timezone: str,
        mdate: str
      }
    }

### 重要な変換仕様

-   alternatenames は「カンマ区切り文字列」→ list\[str\] に変換（必須）
-   latitude / longitude は丸めず float 型で保持
-   geonameid を唯一の識別子とする
-   name / asciiname / alternatenames は正規化前の原文を保持

------------------------------------------------------------------------

## 4. 副インデックス設計（検索最適化）

高速検索のため、以下の副インデックスを同時生成する。

### by_ccode

    "JP" → [geonameid, ...]

用途： - 国単位処理 - 海外領土統合処理

### by_fcl

    "T" → [geonameid, ...]

用途： - 自然地名のみ抽出 - 行政単位のみ抽出

### by_fcode

    "PPL" → [geonameid, ...]

用途： - 都市抽出 - 特定種別限定処理

------------------------------------------------------------------------

## 5. 海外領土統合設計

海外領土は別JSONで管理：

    {
      "AU": ["AU","CC","CX","HM","NF"],
      "NZ": ["NZ","TK"]
    }

処理時：

1.  宗主国コードを取得
2.  対象ccode群を解決
3.  by_ccode を使って geonameid を統合取得

------------------------------------------------------------------------

## 6. 列名短縮設計

内部処理効率向上のため短縮名を採用：

  |原列名|内部キー|
  | ---- | ---- |
  |latitude|lat|
  |longitude|lon|
  |feature_class|fcl|
  |feature_code|fcode|
  |country_code|ccode|
  |admin1 code|ad1|
  |admin2 code|ad2|
  |admin3 code|ad3|
  |admin4 code|ad4|
 | modification date|mdate|

※ データ意味は維持する。

------------------------------------------------------------------------

## 7. マッチング・距離フェーズとの接続

-   正規化は name / asciiname / alternatenames に対して実施
-   マッチ結果は geonameid で保持
-   距離計算は lat / lon を直接使用
-   GeoPandas DataFrame化を前提とする

------------------------------------------------------------------------

## 8. 設計思想

本構造は辞書ではなく、

-   by_id → 主キー（オブジェクト本体）
-   by_ccode / by_fcl / by_fcode → 副インデックス（検索経路）

という二層構造を持つ。

これは軽量RDBのような設計思想であり、
今後の拡張に耐えるデータ基盤である。
