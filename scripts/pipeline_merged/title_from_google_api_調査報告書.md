# title_from_google_api 調査報告書

## 1. 概要

GeoJSON データ（`geojson/by_region/`）に格納されている `title_from_google_api` について、データ形式・取得方法・既存マッチングとの関連・今後の活用方針を調査した結果をまとめる。

---

## 2. プロジェクトの背景と経緯

### 2.1 業務の最終目標

世界大地図帳の索引地名（約35,000件）を GeoNames の ID と紐付けること。GeoNames ID が確定すれば:

- 緯度経度が取得できる
- 各言語の正式な地名表記が取得できる
- 地名の経年変化（名称変更・消滅・移動など）に自動対応する仕組みを構築できる

1ヶ月後に自動更新の仕組みを担当するプログラマーが始動予定であり、それまでに GeoNames ID の紐付け精度を可能な限り高めることが当面の業務目標となる。

### 2.2 データの成り立ち

| 時期 | 出来事 |
|------|--------|
| 2〜3年前 | 前任者が世界大地図帳の索引データを KML 形式で整備。KML → GeoJSON に変換し、さらに Excel 化。Google API（逆ジオコーディング）で `title_from_google_api` も取得した |
| その後 | 紙の地図（世界大地図帳）に地名の追加・編集が行われた。**紙の地図のため緯度経度という概念がなく、追加された地名には座標が存在しない** |
| 現在の業務開始時 | GeoJSON データと Excel データの両方が渡されたが、Excel データを中心に地名マッチング作業を進めていた |

### 2.3 既存データの活用に至るまでの経緯

当初は Excel 上の地名（「修正後欧文地名」列）と GeoNames の完全一致マッチングで進めていたが、マッチング取得率が想定より低く行き詰まっていた。

その後、会社側から以下の2点が追加で提供・指示された:

1. **GeoJSON の `title_from_google_api`（Google API 地名）の活用** — アゼルバイジャンを例に調査するよう指示があった
2. **中国・韓国・北朝鮮の会社独自データ** — 別途提供された

これらは以前から GeoJSON 内に存在していた、あるいは会社が保有していたデータだが、業務の進行過程でその活用可能性が認識された。初めての業務であったことや、前任者からの引き継ぎが十分でなかったことも重なり、このタイミングでの着手となった。

現在は、これまでの Excel ベースのマッチング結果と、新たに活用可能になった Google API 地名を組み合わせて、マッチング精度の向上と補完を進めている段階にある。

### 2.4 0マッチ地名に座標がない理由

0マッチ（×）の 2,263件は、GeoJSON・Excel ともに座標が null である。これは:

- 紙の地図上で追加・編集された地名であり、**そもそも KML に登録されていない**（KML が存在しない）
- KML に登録されていないため、Google API（逆ジオコーディング）も呼べなかった
- 結果として `title_from_google_api` = None、`geometry.coordinates` = `[null, null]`

つまり、「Google API から情報を取得できなかった」のではなく、「KML に存在しなかったため、Google API を呼ぶ手段がなかった」が正確な説明である。

## 3. 前提知識: KML とは

### 3.1 KML の定義

**KML（Keyhole Markup Language）** は、Google Earth や Google Maps で使われる地理データのファイル形式。XML ベースで、地点の名前・緯度経度・説明文・カテゴリなどを格納する。拡張子は `.kml`。

```xml
<!-- KML のイメージ（簡略化） -->
<Placemark>
  <name>アイウド</name>
  <description>都市 / ルーマニア</description>
  <Point>
    <coordinates>23.717382,46.309431</coordinates>
  </Point>
</Placemark>
```

もともと Google Earth の前身である Keyhole 社が開発したフォーマットで、現在は OGC（Open Geospatial Consortium）の国際標準。Google Earth で開くと地図上に地点がピン表示される。

### 3.2 本プロジェクトにおける KML

前任者が世界大地図帳の索引データを KML 形式で管理していた。各地点には以下の情報が含まれていた:

- 地名（日本語・欧文）
- 緯度経度
- カテゴリ（都市・山・川など）
- 国コード
- 読み仮名

この KML データが GeoJSON に変換されたのが、現在の `geojson/by_region/*.geojson` である。GeoJSON の properties に `kml_ID`, `kml_name`, `kml_yomi`, `kml_country`, `kml_category` などの項目があるのは、**元データが KML であった名残** である。

### 3.3 KML と GeoJSON の違い

| 項目 | KML | GeoJSON |
|------|-----|---------|
| 形式 | XML ベース | JSON ベース |
| 主な用途 | Google Earth / Google Maps | Web アプリ / プログラム処理 |
| 拡張子 | `.kml` | `.geojson` |
| 構造 | `<Placemark>` タグ | `Feature` オブジェクト |

どちらも地点の名前・座標・属性を格納する点は同じ。プログラムで処理するには GeoJSON のほうが扱いやすいため、前任者が変換したと推定される。

## 4. title_from_google_api の正体

### 4.1 何をしたのか — 逆ジオコーディング

前任者は **KML の緯度経度を Google API に送信し、「その座標の近くにある地名」を取得した**。これを **逆ジオコーディング（Reverse Geocoding）** と呼ぶ。

| 用語 | 入力 | 出力 | 例 |
|------|------|------|---|
| ジオコーディング | 地名・住所 | 緯度経度 | "東京駅" → 35.681, 139.767 |
| **逆ジオコーディング** | **緯度経度** | **地名・住所** | 35.681, 139.767 → "東京駅" |

つまり「この座標の近くにある地名は何ですか？」と Google に聞き、Google が回答した地名が `title_from_google_api` である。

**地名を入力しているのではなく、緯度経度を入力している** 点が重要。KML の地名（索引データの地名）は Google API の入力には使われていない。

### 4.2 取得の流れ

```
① KML の緯度経度を取得
   例: 緯度=46.309, 経度=23.717（アイウドの座標）

② Google API（逆ジオコーディング）に緯度経度を送信
   → 「この座標の最寄りの地名は何ですか？」

③ Google API の回答
   {
     "compound_code": "H2G4+X3X Aiud, Romania",
     "global_code": "8GF7G275+PJR"
   }
   → 「この座標の最寄りの地名は Aiud（ルーマニア）です」

④ 回答を GeoJSON の properties に格納
   "title_from_google_api": {"compound_code": "...", "global_code": "..."}
```

### 4.3 Google API 地名の信頼度

KML の緯度経度を使って逆ジオコーディングしているため、**地名だけで検索するよりも「正しい地名」を得やすい**。

理由:
- Google は緯度経度から最寄りの地名を返す。つまり **「その座標に実際に存在する地名」** を返している
- 索引データの地名には誤植やローマ字表記の揺れがあるが、Google API はそれに影響されない
- Google は独自のデータベースで地名を管理しており、**現地で実際に使われている地名** である可能性が高い

**ただし重要な前提条件がある:**

| KML の座標が… | Google API の結果 | 信頼度 |
|---|---|---|
| **正しい** | その地点の正しい地名が返る | **高い** |
| **間違っている** | 間違った座標の近くの別の地名が返る | **低い** |

Google API の地名の信頼度は **KML の元座標の精度に依存する**。「会社の緯度経度は間違いが多い」という事情があるため、一部の Google API 地名も、元座標の誤りに引きずられて別の地名が返っている可能性がある。

### 4.4 使用された API の特定

Google Maps Platform の **Geocoding API（逆ジオコーディング）** で取得されたと推定される。

Plus Code（`global_code`, `compound_code`）は別の API ではなく、**Geocoding API のレスポンスに自動的に含まれる情報** である。つまり、前任者は 1種類の API（逆ジオコーディング）を呼んだだけで、Plus Code も住所も同時に取得している。

```
前任者の操作（1回の API 呼び出し）:
  KML の緯度経度 → Google Geocoding API に送信

Google API のレスポンス（全部まとめて返ってくる）:
  {
    "plus_code": {                                    ← 自動的に含まれる
      "global_code": "8GF7G275+PJR",
      "compound_code": "G275+PJR Agios Efstratios, Greece"
    },
    "results": [
      {
        "formatted_address": "Agios Efstratios 815 00, Greece",
        "geometry": { "location": { "lat": 39.514, "lng": 25.009 } }
      }
    ]
  }
```

前任者はこのレスポンスから:
- `plus_code` 部分を保存 → dict 形式（compound_code + global_code）のデータ
- `formatted_address` 部分を保存 → 文字列形式のデータ

のどちらかを格納したと推定される。保存方法が地点によって異なる理由は、取得を複数回に分けて行い、スクリプトの保存ロジックが変更されたためと考えられる。

### 4.5 Plus Code とは

Plus Code（Open Location Code）は、Google が開発した位置を表すコード体系。住所がない地域でも位置を特定できるように設計されている。

```
例: 8GF7G275+PJR
     ^^^^^^^^ ^^^
     エリア   ローカル
```

- `global_code`: 世界中で一意のコード（例: `8GF7G275+PJR`）
- `compound_code`: ローカル部分 + 最寄りの地名（例: `G275+PJR Agios Efstratios, Greece`）

`compound_code` に含まれる地名部分（`Agios Efstratios`）が、地名マッチングに利用できる情報である。

### 4.6 取得にかかった時間の推定

| 項目 | 数値 |
|------|------|
| API 呼び出し対象 | 約 32,649件（none の 2,266件を除く） |
| Google API レート制限（有料プラン） | 通常 50 リクエスト/秒 |
| 推定所要時間（有料プラン） | 約 11〜15分 |
| 推定所要時間（無料枠 10 req/秒） | 約 55分 |

エラー処理・リトライ・ログ出力等を含めると、1〜2時間程度で完了したと推定される。

## 5. データ形式の分類

`title_from_google_api` は以下の4形式で格納されている。

### 5.1 形式の定義

| 形式 | Python 型 | 内容 | 地名抽出 |
|------|-----------|------|----------|
| **dict + compound_code** | dict | Plus Code + 地名 + 国名 | ○ 可能 |
| **文字列（住所形式）** | str | Google の `formatted_address` に相当する住所文字列 | △ 困難 |
| **dict（global_code のみ）** | dict | Plus Code のみ。近くに地名が見つからなかった | × 不可 |
| **None** | なし | キー自体が存在しない or 値が None | × 不可 |

### 5.2 各形式のデータ例

**dict + compound_code:**
```json
{
  "compound_code": "G275+PJR Agios Efstratios, Greece",
  "global_code": "8GF7G275+PJR"
}
```
→ 地名「Agios Efstratios」が抽出可能

**文字列（住所形式）:**
```
"Strada Cuza Vodă 35, Aiud 515200, Romania"
"Johannisstraße 9, 99817 Eisenach, Germany"
```
→ 番地・郵便番号・通り名が混在し、地名の位置が不定

**dict（global_code のみ）:**
```json
{"global_code": "9VXCQMM5+X49"}
```
→ 砂漠・海上・僻地など、近くに地名がない地点

**None:**
→ 座標が null のため API を呼べなかった（0マッチ地名に集中）

### 5.3 なぜ dict 形式と文字列形式が混在するのか

2種類の API（Plus Codes API と Geocoding API）が併用されたか、取得が複数回に分けて行われ、API のバージョンまたは設定が異なっていたと推定される。

dict 形式の key 構造は以下の2パターンのみで統一されている:

| key 構造 | 件数 |
|----------|-----:|
| `(compound_code, global_code)` | 15,367 |
| `(global_code,)` | 3,085 |

### 6.4 エリア別分布の詳細解説 — 44% は「取得の限界」ではない

#### 各列が意味すること

Google API（逆ジオコーディング）のレスポンスには、**常に複数の情報** が含まれる:

```
Google API のレスポンス（1回の呼び出しで全部返る）
┌──────────────────────────────────────────────────┐
│ plus_code:                                        │
│   global_code: "8GF7G275+PJR"          ← 常にある │
│   compound_code: "G275+PJR Agios..., Greece"      │
│                                ← 近くに地名があればある │
│                                                    │
│ results[0]:                                        │
│   formatted_address: "Agios Efstratios 815 00..."  │
│                                ← ほぼ常にある        │
│   geometry.location: {lat, lng}    ← 常にある       │
└──────────────────────────────────────────────────┘
```

前任者のスクリプトが **レスポンスのどの部分を保存したか** によって、4つの形式に分かれている:

| 列名 | 保存された内容 | 意味 |
|------|--------------|------|
| **compound_code** | `plus_code` 部分を dict として保存した | `compound_code` が含まれていたので地名抽出可能 |
| **address_string** | `formatted_address` を文字列として保存した | **同じ地点でも `plus_code` は保存されなかった**。本来は compound_code も返されていた可能性が高い |
| **global_code_only** | `plus_code` 部分を dict として保存したが、`compound_code` がなかった | 砂漠・海上など近くに地名がない場所。これだけが Google API の回答自体の限界 |
| **none** | API 自体を呼んでいない | 座標が null のため API を呼べなかった |

#### address_string（14,197件）にも compound_code があった可能性が高い理由

address_string として保存されている地点でも、Google API は `plus_code`（compound_code 含む）を返していたはず。しかし前任者のスクリプトが `formatted_address` だけを保存し、Plus Code 情報を捨てている:

```
実際の API レスポンス（address_string の地点でも）:
  plus_code:
    compound_code: "H2G4+X3X Aiud, Romania"    ← あったが保存されなかった
    global_code: "8GR4H2G4+X3X"                ← あったが保存されなかった
  results[0]:
    formatted_address: "Strada Cuza Vodă 35, Aiud 515200, Romania"
                                                ← これだけ保存された
```

#### エリアごとに差がある理由

差の原因は Google API の性能差ではなく、**前任者がデータ処理を複数回に分けて行い、その間に保存ロジックを変更したため** と推定される:

| エリア | compound_code | address_string | 推定される処理時期 |
|--------|----:|----:|------|
| **アジア** | **63.8%** | 22.1% | 最初に処理。スクリプトが `plus_code` を保存する設定だった |
| **アフリカ** | 42.4% | 44.2% | 中間時期に処理。両方の保存方法が混在 |
| **ヨーロッパ** | 32.3% | **51.6%** | 後から処理。スクリプトが `formatted_address` 保存に変更 |
| **オセアニア** | 32.4% | 45.8% | 離島が多く global_code_only も 18.6% と最多 |
| **北米・中米** | **24.1%** | **62.1%** | 最も後に処理。`formatted_address` 寄りに変更された |

#### 全体像の整理

```
全 34,915件
├── 座標あり（32,649件）→ Google API を呼べた
│   ├── dict で保存（18,452件）
│   │   ├── compound_code あり（15,367件 = 44%）→ 地名抽出可能
│   │   └── compound_code なし（ 3,085件 =  9%）→ 僻地で地名なし（APIの限界）
│   └── 文字列で保存（14,197件 = 41%）
│       └── ここにも compound_code があった可能性が高いが、
│           formatted_address だけ保存され、Plus Code 情報は失われた
└── 座標なし（ 2,266件 =  6%）→ API を呼べなかった
```

**結論: compound_code 44% は Google API の取得限界ではなく、保存方法の偏りである。** 座標がある 32,649件に対して再度 API を呼べば、僻地の約3,000件を除く約29,000件で compound_code（地名付き）が取得できるはずである。

## 7. Google API のコスト

### 7.1 料金体系

Google Maps Platform の API は **従量課金制** である。

| 項目 | 内容 |
|------|------|
| 基本構造 | 1回の API 呼び出しごとに課金 |
| 無料枠 | 毎月 $200 分のクレジットが付与される（Google Cloud アカウントに紐づく） |
| 逆ジオコーディング単価 | **$5.00 / 1,000リクエスト**（1件あたり約0.75円） |

### 7.2 本プロジェクトでの費用見積もり

| シナリオ | 件数 | 費用（税別） |
|----------|-----:|---:|
| address_string の再取得 | 14,197件 | 約 $71（約10,600円） |
| 0マッチの新規取得 | 2,263件 | 約 $11（約1,700円） |
| 全件再取得 | 32,649件 | 約 $163（約24,500円） |

金額自体は大きくない（全件でも約2.5万円、毎月の無料枠 $200 以内なら実質0円）。

### 7.3 費用よりもハードルが高いもの

API を再取得する場合、費用以上に以下の手続き・環境構築が必要になる:

| 必要なもの | 説明 |
|------------|------|
| Google Cloud アカウント | 課金を有効化する必要がある（クレジットカード登録） |
| API キーの発行 | Google Cloud Console で Geocoding API を有効化し、API キーを生成 |
| セキュリティ管理 | API キーの漏洩防止（.env ファイルでの管理、リファラ制限など） |
| 社内承認 | 会社のセキュリティポリシーにより、API キー取得に承認が必要な場合がある |

前任者が使っていた API キーが現在も有効かどうかは不明。

### 7.4 API 不要な施策との対比

| 方法 | API 呼び出し | 費用 | 環境構築 |
|------|---|---|---|
| compound_code → 地名抽出 | **不要** | **0円** | **不要**（Python のみ） |
| Plus Code → 緯度経度変換 | **不要** | **0円** | **不要**（Python ライブラリのみ） |

## 11. Google API から緯度経度が取得できていない件

### 11.1 現状

`title_from_google_api` には緯度経度が含まれていない。含まれているのは:
- Plus Code（`global_code`, `compound_code`）
- 住所文字列（`formatted_address` 相当）

前任者が使用した API のレスポンスから、Plus Code / 住所部分のみを保存し、`geometry.location.lat/lng` を保存しなかったと推定される。

### 11.2 Plus Code → 緯度経度変換は「新しい座標」ではない

Plus Code をデコードして緯度経度に変換することは技術的に可能だが、**得られる座標は KML の元座標とほぼ同じ値** である。

```
KML の緯度経度（例: 46.309431, 23.717382）
    ↓ Google API に送信（逆ジオコーディング）
Plus Code が返る（例: 8GR4H2G4+X3X）
    ↓ デコード（Python ライブラリで変換）
緯度経度（例: 46.3094, 23.7174）  ← 元に戻るだけ（グリッド精度で丸められる）
```

これは同じデータの往復であり、「Google が独自に判定した新しい座標」ではない。KML 座標 → Plus Code → 緯度経度は、変換精度の丸め（約14m×14m）を除けば同一の値に戻る。

したがって、Plus Code デコードの実用価値は限定的:
- KML 座標が既にある地点（32,649件）→ Plus Code デコードは不要（既に同じ座標がある）
- KML 座標が null の地点（2,266件）→ Plus Code もない（`title_from_google_api` = None のため）

### 11.3 座標データの全体像

本プロジェクトの座標データは **全て同一の出所（KML）** である:

| データの場所 | 内容 | 出所 |
|---|---|---|
| GeoJSON `geometry.coordinates` | KML の緯度経度そのもの | KML |
| Excel `local_lat`, `local_lon` | KML の緯度経度そのもの | KML |
| Plus Code（`global_code`） | KML の緯度経度を元に Google API が生成 | KML |

3つとも同じ KML 座標に由来しているため、どれを使っても同じ情報しか得られない。

### 11.4 0マッチ地名の座標問題

0マッチ（×）の 2,263件は:
- GeoJSON の座標 = `[null, null]`
- Excel の緯度経度 = KML と同一なので、こちらも null
- `title_from_google_api` = None（座標 null のため API を呼べなかった）

つまり **0マッチ地名には、どのデータソースにも座標が存在しない**。

### 11.5 0マッチ地名の座標を新たに得る方法

0マッチ地名に座標を付与するには、KML 以外の情報源が必要:

| 方法 | 内容 | 実現性 |
|------|------|--------|
| GeoNames マッチ結果の座標を流用 | 司令塔②等であいまいマッチした結果の GeoNames 座標を使用 | △ マッチできた地名のみ |
| 地名で Google ジオコーディング（正方向） | 地名を入力 → Google API が緯度経度を返す | ○ API 費用が必要 |
| 手動入力 | Google Maps 等で目視確認し座標を付与 | × 2,263件は非現実的 |

「地名で Google ジオコーディング（正方向）」は、これまでの逆ジオコーディング（座標→地名）とは**逆方向**の処理であり、座標がなくても地名さえあれば実行可能。ただし API 費用と環境構築が必要（第6章参照）。

## 13. 打合せ用まとめ

### プロジェクトの目標

- **最終目標**: 世界大地図帳の索引地名（約35,000件）に GeoNames ID を紐付ける
- **GeoNames ID が確定すれば**: 緯度経度・正式な地名表記の取得、経年変化への自動対応が可能になる
- **1ヶ月後**: 自動更新の仕組みを担当するプログラマーが始動予定

### 現状の把握

- Excel ベースの地名マッチング（司令塔①②）は実施済み。マッチング率は想定より低かった
- GeoJSON の `title_from_google_api` の活用が新たに開始された段階
- `title_from_google_api` は KML の緯度経度を Google API（逆ジオコーディング）に送信して取得した地名情報
- 全 34,915件中、**地名抽出可能は 15,367件（44%）**。ただしこれは API の限界ではなく保存方法の偏り
- 0マッチ（×）の 2,263件は KML に未登録のため、Google API データが存在しない
- **座標データは KML・Excel・Plus Code の全てが同一出所**（KML）であり、Plus Code デコードで新たな座標は得られない

### 提案事項

| 優先度 | 施策 | コスト | 期待効果 |
|--------|------|--------|----------|
| **高** | compound_code → GeoNames マッチング（実装済み） | 0円 | 既存マッチとの比較・補完 |
| 中 | address_string からの地名抽出（Phase2） | 0円 | カバー率を 44% → 85% に向上 |
| 中 | 0マッチ地名の正方向ジオコーディング | API 費用 | 地名から座標を取得。Google API の地名も同時に得られる |
