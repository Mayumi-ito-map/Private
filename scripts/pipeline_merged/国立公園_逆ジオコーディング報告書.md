# 国立公園 逆ジオコーディング作業報告書

## 1. 概要

0マッチ地名の中から「国立公園」に該当する 78 件を対象に、座標（`work_lat` / `work_lon`）を用いた外部 API 照会を行い、地名情報の取得を試みる作業である。

### 目的

- Google Geocoding API の**逆ジオコーディング**で、座標に対応する住所・地名・Plus Code を取得する
- GeoNames API の **nearby search** で、座標近傍の GeoNames レコード（geonameId・名称・カテゴリ等）を取得する
- これらの結果を組み合わせ、国立公園名と GeoNames ID の紐付けの手がかりを得る

### 背景

国立公園は一般的な地点（都市・集落・山など）とは異なり、GeoNames 上では保護区域（`fcl=L`, `fcode=RESN` 等）に分類されることが多い。司令塔①（`run_local_build.py`）のカテゴリフィルタ（P, A, S, T のみ通過）では取得できない可能性が高く、通常のパイプラインとは別のアプローチが必要であった。

## 3. ファイルサイズ問題（ゴースト書式）

### 問題

`国立公園.xlsx` のファイルサイズが **26 MB** であった。データは 79 行しかないにもかかわらず、異常に大きい。

### 調査結果

| 項目 | 値 |
|------|-----|
| Excel 内部の max_row | **1,017,087 行** |
| Excel 内部の max_column | 12 列 |
| 実際にデータがある行 | **79 行** |
| 空行（書式のみ） | **1,017,008 行** |
| `xl/worksheets/sheet1.xml` 非圧縮サイズ | **212 MB** |

### 原因

**「ゴースト書式」** と呼ばれる現象。列全体や大範囲を選択して書式設定（フォント・罫線・背景色など）を適用した結果、データが無い約 100 万行分のセル定義が XML に記録されていた。

新しい Excel にコピー＆ペーストしてもサイズが変わらない理由は、書式ごとコピーされるためである。

### 解決方法

データ部分（79 行）のみを「値のみ貼り付け」で新規ファイルに移すことで解決。

### 今後の予防策

| 操作 | 方法 |
|------|------|
| 貼り付け時 | 「値のみ貼り付け」（Cmd+Shift+V）を使用し、書式を持ち込まない |
| 確認方法 | Cmd+↓ で「Excel が認識している最終セル」に飛び、データ最終行と一致するか確認 |
| 修正方法（Excel 上） | データ最終行の次の行から Ctrl+Shift+End で末尾を選択 → 削除 → 上書き保存 |

## 4. 作成スクリプト

### ファイル

`scripts/pipeline_merged/run_google_reverse_geocoding_parks.py`

### 機能

1 つのスクリプトで以下の 2 つの API を同時に呼び出す:

| API | サービス | 取得内容 |
|-----|----------|----------|
| Google Geocoding API | 逆ジオコーディング | 住所・place_id・Plus Code・座標 |
| GeoNames findNearbyJSON | 近傍検索 | geonameId・地名・国コード・カテゴリ・距離 |

### 実行方法

```bash
# 環境変数の設定（事前に 1 回）
export GOOGLE_MAPS_API_KEY="YOUR_API_KEY"
export GEONAMES_USERNAME="YOUR_USERNAME"

# 実行
cd /Users/itoumayumi/Desktop/project/scripts/pipeline_merged
python run_google_reverse_geocoding_parks.py
```

### 主要オプション

| オプション | デフォルト | 説明 |
|------------|-----------|------|
| `--input` | `0match/国立公園.xlsx` | 入力 Excel |
| `--output` | `0match/国立公園_google_reverse_geocoding.xlsx` | 出力 Excel |
| `--api-key` | 環境変数 `GOOGLE_MAPS_API_KEY` | Google API キー |
| `--geonames-username` | 環境変数 `GEONAMES_USERNAME` | GeoNames ユーザー名 |
| `--language` | `en` | Google API の言語パラメータ |
| `--sleep-sec` | `0.1` | リクエスト間の待機秒数 |
| `--geonames-radius-km` | `30.0` | GeoNames 近傍検索の半径（km） |
| `--geonames-max-rows` | `10` | GeoNames 近傍検索の取得件数上限 |
| `--skip-existing` | OFF | 既存結果がある行をスキップ |

### RAW JSON 保存

各リクエストの生レスポンスを個別の JSON ファイルとして保存:

| フォルダ | 内容 |
|----------|------|
| `0match/google_reverse_geocoding_raw/` | Google API レスポンス（`{index_ID}.json`） |
| `0match/geonames_nearby_raw/` | GeoNames API レスポンス（`{index_ID}.json`） |

RAW JSON を保存する理由:
- Excel には最上位 1 件のみ記載するが、GeoNames nearby は最大 10 件返却される。JSON を参照すれば 2 位以降の候補も確認できる
- 後から別の情報を抽出する際に API を再呼び出しする必要がない（コスト節約）

## 6. API 設定手順

### 6.1 Google Geocoding API

#### 有効化手順

1. [Google Cloud Console](https://console.cloud.google.com/) にアクセス
2. プロジェクトを選択（または新規作成）
3. 左メニュー「API とサービス」→「ライブラリ」
4. **「Geocoding API」** を検索して選択し、「有効にする」をクリック
5. 「認証情報」→「認証情報を作成」→「API キー」で API キーを取得

#### 料金

| 項目 | 内容 |
|------|------|
| 課金モデル | 従量制（pay-as-you-go） |
| 月間無料枠 | $200 分（約 40,000 リクエスト相当） |
| 逆ジオコーディング単価 | $5.00 / 1,000 リクエスト |
| 78 件の概算コスト | 約 $0.39（無料枠内） |

### 6.2 GeoNames API

1. [GeoNames](https://www.geonames.org/) でアカウント登録
2. [アカウント管理ページ](https://www.geonames.org/manageaccount) で「Free Web Services」を有効化
3. ユーザー名を環境変数 `GEONAMES_USERNAME` に設定

#### 料金

無料（1 日あたり最大 20,000 リクエスト）

### 6.3 Mac での環境変数設定

ターミナルで以下を実行:

nano ~/.zshrc

# 3. 設定を反映

# 4. 確認

## 7. 処理フロー

```
入力: 0match/国立公園.xlsx
  │
  ├─ work_lat / work_lon が空 → rg_status = SKIP_NO_WORK_COORD（スキップ）
  │
  └─ work_lat / work_lon あり
       │
       ├─ [1] Google Geocoding API: 逆ジオコーディング
       │    ├─ リクエスト: latlng={lat},{lon}&language=en&key={key}
       │    ├─ レスポンス: 住所・place_id・Plus Code・座標
       │    └─ RAW JSON 保存: google_reverse_geocoding_raw/{index_ID}.json
       │
       ├─ [2] GeoNames findNearbyJSON API: 近傍検索
       │    ├─ リクエスト: lat={lat}&lng={lon}&radius=30&maxRows=10&style=FULL
       │    ├─ レスポンス: 近傍の地名一覧（最大 10 件）
       │    └─ RAW JSON 保存: geonames_nearby_raw/{index_ID}.json
       │
       └─ 0.1 秒の待機（API レート制限対策）

出力: 0match/国立公園_google_reverse_geocoding.xlsx
  ├─ sheet: all（全件）
  ├─ sheet: ok（Google OK のみ）
  ├─ sheet: not_ok（Google 非 OK）
  └─ sheet: geonames_ok（GeoNames OK のみ）
```

## 8. 国立公園特有の留意点

### 8.1 GeoNames でのカテゴリ

国立公園は GeoNames 上で以下のカテゴリに分類されることが多い:

| fcl | fcode | 説明 |
|-----|-------|------|
| L | RESN | 自然保護区（Nature Reserve） |
| L | RESW | 野生生物保護区（Wildlife Reserve） |
| L | PRK | 公園（Park） |
| T | — | 地形として登録されている場合もある |

司令塔①のカテゴリフィルタ（P, A, S, T）では `fcl=L` は通過しないため、通常パイプラインでは取得できない。

### 8.2 GeoNames nearby の制約

`findNearbyJSON` は座標からの距離が近い順に返却する。最近傍が国立公園のレコードとは限らず、近くの集落や川が先に来ることがある。RAW JSON の 2 位以降の候補を手動で確認する運用が必要。

### 8.3 他の情報源

国立公園の情報は GeoNames 以外にも取得可能な情報源がある:

| 情報源 | 内容 | URL |
|--------|------|-----|
| WDPA（World Database on Protected Areas） | 世界の保護区域データベース | https://www.protectedplanet.net/ |
| OpenStreetMap（Nominatim / Overpass） | 地図データから保護区域を検索 | https://nominatim.openstreetmap.org/ |
| Wikidata | 構造化データ（GeoNames ID との紐付けあり） | https://www.wikidata.org/ |
| 各国の国立公園管理機関 | 公式サイトから名称・座標を取得 | 国により異なる |
