### 2.3 司令塔②を通すもの・通さないものの判断根拠

#### ■ 司令塔②を通すべきもの（正規化マッチング対象）

| 対象 | 理由 |
|------|------|
| ① 司令塔①で `judge==0` の地名 | 完全一致で GeoNames にヒットしなかった地名のみが対象。ダイアクリティカル（Ş → S、ä → a 等）やスペルの揺れを正規化（Stage1〜3）することで、マッチする可能性がある。これが司令塔②の本来の目的である |

**判断のポイント:**

1. **地名ソースが同一**（local の「修正後欧文地名」）で、マッチ手法だけを変える（完全一致 → 正規化一致）という正当な次ステップである
2. **国コードが確定している**（`cc` 列）ので、国別に GeoNames を絞り込んで正規化マッチングが安全に行える
3. **正規化の対象は「GeoNames 側のスペル揺れ」**であり、ローカル地名の曖昧さに起因する誤マッチではない

---

#### ■ 司令塔②を通す必要がないもの（フラグのみ付与）

**④ 全世界マッチング**

| 観点 | 説明 |
|------|------|
| 目的 | 国境沿いの山名・峠名、海名・湾名など、**国で分類されていない地名**を隣接国やグローバルデータから探す |
| 司令塔②に通さない理由 | 全世界マッチングは「検索範囲を広げる」ことが目的であり、「スペル揺れの吸収」が目的ではない。全世界データに対して正規化を行うと、意図しない国・カテゴリの地名に誤マッチするリスクが高い |
| 具体例 | 例えば `Mont Blanc` を全世界で正規化すると、アフリカの `Montblanc` 等にもヒットしうる。国境付近の地名は完全一致で確認できれば十分である |
| フラグとしての価値 | 司令塔① で 0マッチだった地名が全世界マッチングでヒットしていれば、「隣接国のデータにある」という手がかりになる |

**⑤ フィルタなしマッチング（base）**

| 観点 | 説明 |
|------|------|
| 目的 | カテゴリフィルタ（FCL）と距離フィルタを外して、**名前は一致するがフィルタで弾かれたケース**を発見する |
| 司令塔②に通さない理由 | フィルタなしマッチングの本質は「フィルタの有無による差分を確認する」こと。完全一致でヒットした情報こそが「フィルタが原因で弾かれた」証拠になる。正規化を適用すると「スペル揺れで新たにヒットした」ケースと区別できなくなり、差分分析の意味が失われる |
| 具体例 | AZE の調査で、`Ordubad`（gid=147368）は名前の完全一致でヒットしていたが距離フィルタで弾かれていた。フィルタなしの結果で `base=1` のフラグがあれば、「名前は正しく、フィルタ条件の見直しが必要」と判断できる |
| フラグとしての価値 | `final_status==zero` かつ `base=1` であれば、「地名は GeoNames に存在する。ローカル側のカテゴリまたは座標の誤りが原因」と特定できる |

**② Google API マッチング**

| 観点 | 説明 |
|------|------|
| 目的 | ローカルの「修正後欧文地名」とは**独立した地名ソース**（逆ジオコーディングで取得した地名）で GeoNames をマッチングする |
| 司令塔②に通さない理由 | Google API 地名はすでに Google が正規化済みの地名であり、ダイアクリティカルの揺れはほぼ解消されている。これを更に正規化しても得られる改善は少ない。一方で、Google API 地名は Plus Code の `compound_code` から抽出しており、住所の一部（地域名）が入る場合がある。正規化すると関係のない GeoNames レコードに誤ヒットしやすい |
| 具体例 | `CVXF+QW7, Astara, Azerbaijan` → 抽出した `Astara` は Google 側で正規化済み。これを Stage2（子音のみ）にすると `Astar`、`Astur` 等にもヒットする |
| フラグとしての価値 | ローカル地名とは異なるソースでのマッチング結果であり、「ローカル地名は 0マッチだが Google API 地名ではヒットした」場合、**ローカル地名に誤植がある**可能性を示唆する |

---

#### ■ まとめ: 判断フローチャート

```
各データソースの 0マッチ地名
    │
    ├── ① 司令塔①の 0マッチ
    │     → 司令塔②に通す（正規化マッチング）
    │     理由: 同一地名ソース × 国コード確定 × スペル揺れ吸収が目的
    │
    ├── ④ 全世界の 0マッチ
    │     → 司令塔②に通さない（フラグのみ）
    │     理由: 検索範囲の拡大が目的 × 正規化で誤マッチリスク大
    │
    ├── ⑤ フィルタなしの 0マッチ
    │     → 司令塔②に通さない（フラグのみ）
    │     理由: フィルタ差分確認が目的 × 正規化で差分分析の意味が消失
    │
    └── ② Google API の 0マッチ
          → 司令塔②に通さない（フラグのみ）
          理由: 独立ソース × Google 側で正規化済み × 誤マッチリスク
```

## 4. 統合スクリプト: merge_all_results.py

### 4.1 概要

| 項目 | 内容 |
|------|------|
| スクリプト | `scripts/pipeline_merged/merge_all_results.py` |
| 結合キー | `index_id`（Excel は `index_ID` の場合もあり、自動変換） |
| 出力先 | `output_merged/merged_all.xlsx` |

### 4.2 処理の流れ

```
1. ベースデータ読み込み
   output_match_alternatenames_cate/*.xlsx（sheet: all）
   → 全行の基本情報 + 司令塔①の結果

2. 司令塔②の結果を結合
   output_match_results/*_result.xlsx
   → stage_matched, stage_matched_stage 列を追加

3. 補完データを結合
   output_google_api_match/*.xlsx → ga_judge, ga_matched_geonameid 等
   output_match_worldwide_distance/*.xlsx → ww_judge, ww_geonameid 等
   output_match_alternatenames_base/*.xlsx → base_judge, base_geonameid 等

4. final_status の算出
5. 集計表の作成
6. Excel 出力（7シート）
```

### 4.3 final_status の決定ロジック

```
本流で判定（補完データは使わない）:
  司令塔①で judge==1    → hit-1
  司令塔①で judge==2+   → hit-2+
  司令塔②でヒット       → hit-stage1 / hit-stage2 / hit-stage3
  いずれもヒットなし     → zero
```

### 4.4 supplement_flags（補完フラグ）

`final_status == "zero"` の行について、補完データにヒットがあればフラグを立てる:

| フラグ | 意味 |
|--------|------|
| `ww=1` / `ww=2+` | 全世界マッチングでヒットあり |
| `base=1` / `base=2+` | フィルタなしでヒットあり |
| `ga=1` / `ga=2+` | Google API でヒットあり |

例: `supplement_flags = "base=1 / ga=1"` → フィルタなし・Google API の両方でヒットあり

### 4.5 出力 Excel のシート構成

| シート | 内容 |
|--------|------|
| `summary` | 集計表（hit-1 / hit-2+ / stage / zero の件数と割合） |
| `all` | 全行データ |
| `hit-1` | 司令塔①で1件確定 |
| `hit-2+` | 司令塔①で複数マッチ |
| `stage_hit` | 司令塔②でヒット |
| `zero_supplement` | 0マッチだが補完データにヒットあり |
| `zero_unknown` | **完全不明**（全手法で0マッチ） |

## 7. index_id 重複の調査結果（2026-03-31）

### 7.1 調査の経緯

`merge_all_results.py` の初回実行で、統合後の行数がベースデータより多くなった。原因を調査したところ、`index_id` が重複するデータが存在し、`merge`（left join）時に行が膨張していたことが判明した。

調査スクリプト: `scripts/pipeline_merged/check_duplicate_index_id.py`

### 7.2 調査結果の概要

| データソース | 重複の有無 | 備考 |
|---|---|---|
| **入力 Excel（excel_local_merged）** | **あり（根本原因）** | 全7ファイルで計 474 件の重複 index_ID |
| **GeoJSON（geojson/by_region）** | **なし** | 全5エリア 34,915 件、重複ゼロ |

**結論: 重複の根本原因は入力 Excel にある。GeoJSON 側は完全にクリーン。**

### 7.3 入力 Excel のファイル内重複

各エリアファイルの中に、同一 `index_ID` が 2〜6 行存在する。

| ファイル | 行数 | ユニーク | ファイル内重複 ID 数 |
|----------|-----:|--------:|-------------------:|
| cn000_asia.xlsx | 6,091 | 5,940 | 144 |
| cn001_CN_asia.xlsx | 3,318 | 3,315 | 2 |
| cn002_KR-KP_asia.xlsx | 466 | 462 | 4 |
| cn100_europe.xlsx | 9,369 | 9,251 | 110 |
| cn200_africa.xlsx | 3,278 | 3,208 | 61 |
| cn300_america.xlsx | 5,901 | 5,852 | 49 |
| cn450_oceania.xlsx | 3,783 | 3,706 | 71 |
| **合計** | **32,206** | **31,699** | **(延べ 507 行が余剰)** |

重複の例:
- `WJ022252000`（cn100_europe）: 6 行存在
- `WJ001073000`（cn000_asia）: 3 行存在
- `nan`（cn200_africa）: 5 行存在 — index_ID が未付与のデータ

### 7.4 ファイル間重複（エリアをまたぐ重複）

エリアをまたいで同一 `index_ID` が存在するケースは少数だが存在する。

| 重複パターン | 件数 |

### 7.5 各パイプライン出力への波及

入力 Excel の重複が各パイプラインの出力にそのまま引き継がれている。

| パイプライン出力 | 合計行数 | ユニーク | 重複 ID 数 |
|---|---:|---:|---:|
| ① 司令塔①（cate） | 27,783 | 27,309 | 448 |
| ③ 司令塔② | 27,783 | 27,309 | 448 |

| ④ 全世界 | 1,592 | 1,430 | 155 |
| ⑤ フィルタなし | 27,783 | 27,309 | 448 |

Google API マッチング（GeoJSON 由来）のみ重複がゼロ。

### 7.6 統合時の対処

`merge_all_results.py` で結合前に各データソースの `index_id` 重複を排除する。

**重複排除の優先順位:**

| 優先順位 | judge の値 | 意味 |
|:---:|---|---|
| **1（最優先）** | `"1"` | 1件確定 — GeoNames ID が確定している行を残す |
| 2 | `"2+"` | 複数マッチ — 候補がある行を残す |
| 3 | `"0"` / その他 | マッチなし |

### 7.7 根本対処について

入力 Excel の重複は前任者のデータ設計に起因する。現時点では統合スクリプト側で対処し、入力データそのものの修正は本計画の範囲外とする。ただし、重複の実態を把握しておくことで、以下の場面で活用できる:

- hit-2+ の絞り込み時に、同一 index_id の別行が別の GeoNames ID にヒットしている可能性を考慮
- 将来の入力データクリーニング作業の基礎資料
